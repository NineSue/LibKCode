## 1 . 红黑树对标数据 (N=1,000,000)

**操作**：连续运行 10 轮测试（已预热 5 轮），计算平均值后填入下表。

| 性能指标 (ms)         | STL map (A) | LibKCode (B) | 提升倍率 (A/B) |
| --------------------- | ----------- | ------------ | -------------- |
| **随机查找 (Find)**   | 738.00      | 517.83       | **1.42x**      |
| **随机删除 (Erase)**  | 668.67      | 562.64       | **1.19x**      |
| **随机插入 (Insert)** | 557.68      | 539.52       | **1.03x**      |

> 注意：测试数据详细见 ./templates/十次压测对比数据.xlsx

### 结果分析

- **查找 (Find) 优势巨大**：LibKCode 平均比 STL 快 **42.5%**。这得益于内核红黑树实现的紧凑内存布局和高效的指针操作，无 C++ 模板膨胀开销，在缓存局部性上表现更优。在第 9 次测试中，加速比甚至达到了 **1.88x**。
- **删除 (Erase) 稳定领先**：LibKCode 平均快 **18.8%**。表明内核在节点删除后的重新平衡（Rebalance）逻辑更为精简，旋转操作的开销控制得更好。
- **插入 (Insert) 互有胜负**：虽然单次测试中有快有慢，但平均下来 LibKCode 仍以微弱优势（**3.3%**）胜出。考虑到 `std::map` 是高度优化的模板代码，这一结果充分证明了内核实现的极致效率。

## 3. 排序算法特性对标 (N=1,000,000)

| 算法类型     | 耗时 (ms) | 特性说明                                                     |
| ------------ | --------- | ------------------------------------------------------------ |
| `std::sort`  | 61-63     | C++ 内省排序 (Introsort)，平均性能最优，但最坏情况为 $O(n^2)$ |
| `kcode_sort` | ~140      | 内核原生堆排序 (Heapsort)，**$O(1)$ 空间复杂度**，最坏情况严格 $O(n \log n)$ |

### 结果分析
- **性能差异符合预期**：`std::sort` 作为工业级优化的混合排序算法，在平均情况下远快于堆排序。
- **LibKCode 的核心优势不在“快”，而在“稳”和“省”**：
  - **确定性**：无论输入如何，性能表现稳定，无退化风险。
  - **空间效率**：**不占用额外的栈空间**，避免了递归爆栈的风险，这对于嵌入式系统或深度递归场景至关重要。
  - **内核友好**：该实现直接源自内核，保证了与内核行为的一致性。

## 5. 测试结论

本次基准测试全面验证了 LibKCode 项目的两大核心能力：

1.  **红黑树性能卓越**：在 **100 万** 规模的数据集上，LibKCode 的红黑树实现在 **查找** 和 **删除** 操作上显著优于 C++ STL 的 `std::map`，**插入** 操作也保持了微弱优势。这证明了直接复用内核高质量算法的巨大价值。
2.  **排序实现特色鲜明**：`kcode_sort` 虽然在速度上不及 `std::sort`，但其 **$O(1)$ 空间复杂度** 和 **确定性的最坏情况性能** 使其在特定场景（如资源受限、实时系统）中具有不可替代的优势。

综上所述，LibKCode 成功实现了其设计目标：**安全、高效地将 Linux 内核的成熟算法导出至用户态，为开发者提供了高性能、高可靠的新选择**。
